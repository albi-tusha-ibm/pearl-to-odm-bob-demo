## What Is This Repository?

This repo tells the story of how **IBM Bob** modernizes a legacy mortgage-insurance (MI) underwriting rules engine — nicknamed **"PEARL"** — into a governed, auditable, business-readable **IBM Operational Decision Manager (ODM)** Decision Service.

**MGIC (Mortgage Guaranty Insurance Corporation)** is the scenario backdrop: a leading MI carrier whose underwriting logic has accumulated over years in a proprietary internal rules engine. The legacy system is functional but fragile — rules are scattered, inconsistently named, occasionally duplicated, and difficult for business users to read or change safely.

**IBM Bob** ingests the legacy ruleset, structures it, converts it to ODM, validates parity, and produces governance-ready documentation — all in a repeatable, auditable workflow.

## FAQ

**Q: Can this import into a real ODM instance?**
> The `odm_target/export/` folder contains a descriptive placeholder of the export structure. To import into a live ODM instance, a Rule Designer project would need to be created following the architecture in `decision_service_arch.md`. The design artifacts here are complete enough to guide that implementation. See `odm_target/export/README_export.md` for details.

**Q: How do we change LTV thresholds?**
> In ODM, LTV thresholds are managed in Decision Tables within the `EligibilityRules` project. A Rule Author with appropriate permissions opens the table in Decision Center, modifies the threshold cell, and submits for approval. The change is versioned, reviewed, and promoted through DEV → UAT → PROD. No code deployment is required.

**Q: What happens when a rule conflicts?**
> ODM uses explicit rule priorities and ruleflow sequencing to resolve conflicts deterministically. The `change_log.md` documents every conflict found in the legacy system and how it was resolved. Going forward, the governance process in `governance_and_release.md` ensures conflicts are caught in UAT before reaching production.

**Q: How are decisions explained to borrowers?**
> Each ODM rule fires with a named `reason` message (e.g., "LTV exceeds maximum 97% for this product type"). These messages are returned in the decision response payload and can be surfaced in borrower-facing communications, audit logs, or adverse action notices.

**Q: Is this real MGIC logic?**
> No. All rules, thresholds, and data in this repository are synthetic and illustrative. They are designed to feel realistic for MI underwriting but do not represent MGIC's actual proprietary guidelines.

## Security Considerations

### Demo Credentials Warning

⚠️ **IMPORTANT:** This repository contains demo credentials for illustration purposes only.

The file `tools/invoke_odm_stub.py` includes hardcoded credentials (`admin`/`admin`) to demonstrate ODM REST API authentication. **These credentials must NEVER be used in production environments.**

### Production Security Best Practices

When deploying to production:

1. **Use Environment Variables:**
   ```python
   import os
   odm_username = os.environ.get('ODM_USERNAME')
   odm_password = os.environ.get('ODM_PASSWORD')
   odm_host = os.environ.get('ODM_HOST')
   ```

2. **Implement Secret Management:**
   - Use enterprise secret management solutions (HashiCorp Vault, AWS Secrets Manager, Azure Key Vault)
   - Never commit credentials to version control
   - Rotate credentials regularly

3. **Use Strong Authentication:**
   - Prefer OAuth 2.0 or API keys over Basic Authentication
   - Implement multi-factor authentication (MFA) where possible
   - Use service accounts with minimal required permissions

4. **Enable Secure Communications:**
   - Always use HTTPS/TLS for API communications
   - Validate SSL certificates
   - Use VPN or private networks for sensitive data

5. **Follow Security Policies:**
   - Adhere to your organization's security standards
   - Conduct regular security audits
   - Implement logging and monitoring for all API calls
   - Review and update security practices regularly

### Reporting Security Issues

If you discover a security vulnerability in this demo project, please report it by creating a private security advisory or contacting the maintainers directly. Do not create public issues for security vulnerabilities.
